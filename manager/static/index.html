平台能力
把一次压测从“手工跑脚本、手工看报告”，升级成可调度、可观测、可复盘、可扩展的性能工程平台。
从执行工具升级为容量决策平台
对业务
上线前知道扛不扛得住
用真实压测数据验证容量水位，提前发现 RT 抖动、错误率上升和吞吐瓶颈。
对研发
问题定位有证据
请求日志、接口统计、百分位、历史趋势集中查看，减少靠经验猜瓶颈。
对测试
高频压测可复用
脚本、CSV、环境变量、模板和定时任务沉淀下来，重复场景一键复跑。
对管理
结果能汇报可追踪
报告、趋势、对比和性能基线统一留存，让容量结论可以复盘和对齐。
1
准备压测资产
脚本、CSV 参数、环境变量、鉴权配置和常用模板统一管理，团队不再各自保存一份 JMX。
沉淀：脚本库 + 数据集 + 环境配置
2
自动调度执行
Manager 统一分配施压节点，支持分布式压测、任务排队、定时执行、一键重跑和压力机独占。
扩展：增加 Agent 即可横向扩容
3
过程实时可观测
运行中持续推送 QPS、RT、错误率、线程数和资源水位，异常不是压完才知道。
监控：实时曲线 + 告警通知 + 工具日志
4
产出容量结论
自动汇总请求量、QPS、平均 RT、P90/P99、错误率、接口明细、历史趋势和任务对比。
交付：HTML/PDF 报告 + 趋势看板 + JTL 导入
架构设计
核心思路是“管理面和施压面解耦”：平台负责任务、数据、报告和协作，施压节点专心发流量，结果更稳定也更可信。
前端可视化 + 后端调度 + 分布式执行
01 用户入口
Web UI
同事通过页面创建任务、选择脚本、查看实时曲线和报告，不需要登录机器跑命令。
产出：统一操作入口
02 调度中枢
FastAPI Manager
负责任务编排、节点选择、实时推送、报告查询、权限边界和平台 API。
产出：任务可控可追踪
03 消息通道
Redis PubSub
Manager 和 Agent 通过消息解耦，任务下发、心跳、进度、结果都走标准通道。
产出：节点可横向扩展
04 施压执行
Agent + JMeter
每台施压机只关注执行 JMeter、采集进度和回传结果，多节点并发发压。
产出：真实压力流量
05 数据沉淀
Reports + DB
JTL、HTML 报告、任务元数据、趋势数据统一归档，支持复盘、对比和基线建设。
产出：容量评估依据
为什么结果更可信
管理服务不直接发压，施压资源独立运行，减少报表查询、页面访问、数据库写入对发压机的干扰。
管理面和施压面分离
为什么后续能扩
Agent 节点可以按机器规模横向增加，调度层不用重写；未来也能接入 k6、Locust 等执行引擎。
执行器可插拔，节点可扩容
为什么适合团队协作
脚本、数据、环境、日志、报告都在平台内留痕，压测不依赖某个人本地环境，交接和复盘成本低。
资产沉淀，结论可复盘
