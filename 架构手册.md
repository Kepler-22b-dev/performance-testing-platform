# 性能测试平台 - 系统架构手册

> 版本：3.1 | 更新时间：2026-07-07

---

## 一、平台概述

性能测试平台是一个基于 Apache JMeter 的分布式性能测试平台，提供 Web 界面管理压测任务、分析结果、监控资源，并支持移动端性能监控。

### 核心能力

| 能力 | 说明 |
|------|------|
| 🎯 压测执行 | 分布式 JMeter 压测，支持多 Slave 节点 |
| 📊 结果分析 | TPS/RT/错误率曲线、分布图、百分位分析 |
| 📱 移动端监控 | iOS (pymobiledevice3) + Android (SOLOX) 性能采集 |
| ⏱️ 定时调度 | 间隔执行、单次定时 |
| 🔔 告警通知 | 阈值检测 + Webhook 通知 |
| 📈 趋势分析 | 历史性能趋势、任务对比 |

### 3.3 移动端监控模块

支持 iOS 和 Android 双引擎的移动端性能监控。

#### iOS 监控原理

```
iPhone (USB) → pymobiledevice3 → sysmon process single --userspace
                                       ↓
                                  JSON 数据
                                       ↓
                              CPU: cpuUsage (%)
                              Memory: physFootprint (bytes → MB)
                              PID: pid
```

#### Android 监控原理

```
Android (USB) → SOLOX 服务 (:50001) → HTTP API
                                       ↓
                                  JSON 数据
                                       ↓
                              CPU: cpuUsage (%)
                              Memory: total (MB)
                              FPS: fps (Hz)
                              Network: upload/download (KB)
                              Battery: level (%)
```

#### iOS 监控限制

- 需要 iOS 17+ 支持 `--userspace` 模式（无需 sudo）
- `sysmon process single` 命令首次调用约需 20 秒
- 不支持 FPS、网络、电池等指标
- 进程名匹配需要通过 `CFBundleExecutable` 字段

#### 异步架构

为避免阻塞 FastAPI 事件循环，所有 pymobiledevice3 调用使用 `run_in_executor` 异步执行：

```python
@staticmethod
async def _run_cmd_async(cmd: List[str], timeout: int = 15) -> Optional[str]:
    import asyncio
    return await asyncio.get_event_loop().run_in_executor(
        None, IOSMonitor._run_cmd, cmd, timeout
    )
```

## 七、关键流程

### 7.1 任务执行流程

```
1. 用户创建任务 → Web UI
2. 提交到 API → /api/tasks/quick-run
3. 任务调度器分配节点 → scheduler.py
4. 通过 Redis 下发命令 → jmeter:command 频道
5. Agent 接收命令 → agent/main.py
6. 启动 JMeter 执行 → jmeter_runner.py
7. 实时推送进度 → jmeter:progress 频道
8. 执行完成，收集结果 → jmeter:result 频道
9. 生成报告 → reports/
10. 通知用户 → WebSocket + Webhook
```

### 7.2 移动端监控流程

```
1. 页面加载 → autoDetectDevice()
2. 自动检测设备平台 → /api/mobile/detect
3. 加载设备列表和应用列表 → /api/mobile/apps/{platform}
4. 用户选择应用 → onAppSelect()
5. 点击开始监控 → startMobileMonitor()
6. 定时采集数据 → /api/mobile/collect/all (每5秒)
7. 后端调用 pymobiledevice3/SOLOX
8. 返回 CPU/内存数据
9. 更新 ECharts 图表
```

### 动态压力机扩容

#### 目标

实现 Agent 节点的自动发现、负载感知和动态扩缩容，无需人工干预即可按需扩展施压能力。

#### 三阶段演进路线

**阶段一：Agent 自注册 + 资源池（近期）**

改动最小，基于现有 Redis PubSub 机制扩展：

```
新 Agent 启动
  → 连接 Redis → 自动注册到 jmeter:agents 集合
  → 上报资源指标（CPU/内存/磁盘/端口）到 Redis Hash
  → Manager 心跳检测发现新节点，自动纳入调度池
  → 任务调度时按负载评分选择最优 Agent
  → 空闲超时 → Agent 自动退出
```

涉及模块改动：
- `agent/main.py`：启动时上报资源信息，心跳携带实时指标
- `manager/core/scheduler.py`：`get_available_agent()` 改为按负载评分选择
- `manager/core/node_manager.py`：增加资源感知，支持动态扩缩策略
- `manager/core/auto_scaler.py`（新增）：监控负载，触发扩缩决策
- `common/config.py`：新增 `AGENT_IDLE_TIMEOUT`、`AGENT_MAX_SCALE`、`AGENT_COOLDOWN` 等配置

调度策略：
- 扩容触发：任务排队 > 0 且 当前 Agent 数 < `AGENT_MAX_SCALE`
- 缩容触发：所有 Agent 空闲超过 `AGENT_IDLE_TIMEOUT`
- 冷却时间：扩缩操作间隔 ≥ `AGENT_COOLDOWN`
- 负载评分：`CPU空闲度×0.4 + 内存空闲度×0.3 + 磁盘空间×0.2 + 空闲状态×0.1`

**阶段二：Docker 动态扩缩（中期）**

Agent 容器化，Manager 通过 Docker API 动态创建/销毁容器：

```
Manager 检测到任务排队
  → 调用 Docker API 创建 Agent 容器
  → 新容器自动连接 Redis 注册
  → 任务分配完成
  → 空闲超时 → Manager 调用 Docker API 停止容器
```

涉及模块改动：
- 新增 `Dockerfile.agent`：Agent 镜像打包
- 新增 `manager/core/docker_scaler.py`：Docker API 封装
- `manager/core/scheduler.py`：任务排队时触发扩容

扩缩决策参数：
- 最大实例数：`AGENT_MAX_SCALE`（默认 10）
- 最小实例数：`AGENT_MIN_SCALE`（默认 1）
- 冷却时间：`AGENT_COOLDOWN`（默认 120s）
- 镜像名称：`AGENT_DOCKER_IMAGE`（可配置）

**阶段三：Kubernetes HPA（远期）**

企业级方案，适合大规模团队和云原生环境：

```
Agent K8s Deployment (min=1, max=10)
  → 自定义 Metrics Adapter 暴露 jmeter:queue_length
  → HPA 根据队列长度自动扩缩 Pod
  → Pod 启动时自动注册到 Redis
  → 任务完成后缩容到 min
```

需要的基础设施：
- K8s 集群 + Metrics Server
- 自定义 Metrics Adapter（暴露 Redis 队列长度）
- Agent Docker 镜像
- ConfigMap 管理配置

#### 方案选型建议

| 场景 | 推荐方案 |
|------|----------|
| 当前阶段快速验证 | 阶段一（自注册 + 资源池） |
| 已用 Docker 部署 | 阶段二（Docker 动态扩缩） |
| 企业级大规模 | 阶段三（Kubernetes HPA） |
