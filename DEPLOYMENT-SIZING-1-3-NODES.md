# 性能测试平台 Linux 分布式部署配置评估

> 适用范围：日常压测并发通常低于 1000，业务 QPS 大多数低于 2000，部署规模 1-3 台 Linux 机器。  
> 建议用途：给领导/采购/运维说明为什么需要这些机器配置，以及后续如何扩容。  
> 结论先行：如果预算允许，建议采用 3 台机器；如果预算收紧，2 台机器是可接受的生产起步方案；1 台机器只建议用于演示、开发或小规模验证。

---

## 1. 背景和容量目标

本平台由 Manager、Web UI、Redis、PostgreSQL、Agent、JMeter 执行器、JTL/HTML 报告与 CSV 数据文件组成。压测时真正消耗资源的是施压节点上的 JMeter 进程，管理节点主要负责任务调度、实时状态、结果收集、JTL 解析、报告查看和文件存储。

当前容量目标按以下口径评估：

| 指标 | 日常范围 | 规划口径 |
|---|---:|---:|
| 并发用户数 | 多数低于 1000 | 按 1000 并发预留 |
| 业务 QPS | 多数低于 2000 | 按 2000 QPS 峰值预留 |
| 任务时长 | 常见 10-60 分钟 | 按 30-60 分钟结果文件预留 |
| 机器数量 | 1-3 台 | 推荐 3 台，最低生产 2 台 |
| 网络环境 | 内网压测为主 | 要求与被测系统同机房/同 VPC/同内网 |

这里的“并发”和“QPS”不是简单一一对应关系。若接口响应很快、无思考时间、脚本里每个线程循环请求较多，即使并发不到 1000，也可能达到 2000 QPS；反过来，如果接口响应慢或脚本有等待时间，1000 并发也不一定产生 2000 QPS。因此机器配置需要同时覆盖线程数、请求速率、网络吞吐、结果文件写入四个维度。

---

## 2. 推荐结论

### 推荐方案：3 台 Linux 机器

| 节点 | 数量 | 推荐配置 | 磁盘 | 网络 | 作用 |
|---|---:|---|---|---|---|
| 管理节点 | 1 | 4C/8GB 起步，推荐 8C/16GB | 500GB SSD/NVMe | 千兆内网 | Manager、Web UI、Redis、PostgreSQL、报告/JTL/CSV 存储 |
| 施压节点 | 2 | 每台 8C/16GB | 每台 200GB SSD/NVMe | 千兆内网 | Agent + JMeter，分摊并发和 QPS |

这个方案适合正式落地。两台施压机可以把 1000 并发拆成每台约 500 并发，把 2000 QPS 拆成每台约 1000 QPS，单机不会长期顶满 CPU、内存、网络和磁盘 I/O。管理节点独立后，报告生成、JTL 解析、Web 查询不会抢占施压机资源，压测结果更可信。

### 可接受生产起步：2 台 Linux 机器

| 节点 | 数量 | 推荐配置 | 磁盘 | 网络 | 适用范围 |
|---|---:|---|---|---|---|
| 管理节点 | 1 | 4C/8GB，预算允许用 8C/16GB | 300-500GB SSD/NVMe | 千兆内网 | 管理、调度、Redis、PostgreSQL、报告 |
| 施压节点 | 1 | 8C/16GB | 200GB SSD/NVMe | 千兆内网 | 日常 500-800 并发较稳，1000 并发/2000 QPS 可做短时冲刺 |

这个方案可以满足大多数日常低于 1000 并发、低于 2000 QPS 的场景，但容量余量比 3 台方案少。遇到脚本复杂、断言较多、响应体较大、错误日志开启、大量 CSV 参数化、长时间稳压时，单台施压机更容易成为瓶颈。

### 仅建议开发/演示：1 台 Linux 机器

| 节点 | 数量 | 推荐配置 | 磁盘 | 网络 | 适用范围 |
|---|---:|---|---|---|---|
| 管理+施压合一 | 1 | 8C/16GB | 300-500GB SSD/NVMe | 千兆内网 | 功能演示、小规模验证、低风险压测 |

1 台机器可以跑起来，但不建议作为正式压测环境。原因是 Manager、Redis、PostgreSQL、JMeter、报告生成都在同一台机器上，压测过程中资源相互争抢，可能导致压测结果被发压机自身瓶颈污染。

## 3. 为什么需要这个配置

### 3.1 施压机需要 CPU 余量

JMeter 不是简单地“开线程”就结束了。每个采样请求都要完成参数替换、HTTP 连接处理、响应接收、断言判断、结果落盘、统计汇总等操作。QPS 越高、接口响应越快，JMeter 单位时间内处理的采样结果越多，CPU 压力越高。

对当前业务规模，单台 8C/16GB 施压机可以覆盖多数日常压测，但 1000 并发和 2000 QPS 如果同时出现，建议拆到 2 台施压机上运行。这样每台机器保持约 50%-70% 的资源使用率，避免发压机 CPU 打满后误判被测服务性能。

### 3.2 JMeter 需要足够 JVM 堆内存

JMeter 官方文档明确提示，负载测试需要关注 CPU、内存、网络、OS 调优和 Java 堆内存；JMeter 默认堆内存约 1GB，可能不足以支撑线程数较高或脚本较复杂的测试。正式压测必须使用命令行/非 GUI 模式。

建议按以下范围设置每台施压机的 JMeter 堆内存：

| 单台施压机承载并发 | 建议 JMeter 堆内存 | 机器内存建议 | 说明 |
---|---:|---:|---|
| 100-300 | 2-3GB | 8GB+ | 冒烟、低并发验证 |
| 300-600 | 4-6GB | 16GB | 日常负载测试主力区间 |
| 600-1000 | 6-8GB | 16-32GB | 单机高压，脚本需精简 |
| 1000+ | 不建议单机硬扛 | 多施压机拆分 | 优先横向扩容 |

16GB 内存的施压机通常给 JMeter 4-6GB 比较稳妥，剩余内存留给 OS、Agent、文件缓存、JTL 写入和监控进程。不要把 `-Xmx` 设置到接近物理内存上限，否则 Linux 文件缓存和系统进程没有余量，压测后期更容易抖动。

### 3.3 结果文件和报告需要 SSD/NVMe

JMeter 的 JTL 结果文件会随 QPS 和时长线性增长。粗略估算公式：

```text
JTL大小(GB) ≈ QPS × 持续秒数 × 单条采样字节数 / 1024^3
```

单条采样在精简 CSV 输出时通常可按 0.3-1KB 估算。以 2000 QPS、30 分钟计算：

```text
2000 × 1800 × 0.3KB ≈ 1.0GB
2000 × 1800 × 1.0KB ≈ 3.4GB
```

如果开启错误响应日志、保存更多字段、接口响应体较大、任务保留周期较长，磁盘增长会更快。因此：

- 管理节点建议 500GB SSD/NVMe，用于脚本、CSV、JTL、HTML 报告、数据库和日志保留。
- 每台施压节点建议 200GB SSD/NVMe，用于本地 JMeter 临时文件、JTL、错误日志、任务日志。
- 不建议用机械盘作为主要结果文件盘，长时间高 QPS 写入和报告生成会受 I/O 影响。

### 3.4 网络需要留足带宽

压测流量来自施压机，而不是管理节点。若接口响应体较小，例如平均请求+响应 5-20KB，2000 QPS 大致会产生 80-320Mbps 量级流量，千兆内网一般够用。若响应体达到 50-100KB，2000 QPS 可能接近或超过千兆网卡舒适区，应考虑：

- 增加施压机数量，分摊网络吞吐。
- 使用 10Gbps 内网。
- 减少响应体保存和错误日志保存范围。
- 确保施压机和被测系统在同一机房、同一 VPC 或同一低延迟网络。

JMeter 官方也建议施压节点与被测应用放在同一以太网段附近，但不要部署在应用服务器本机上，否则会污染测试结果。

### 3.5 管理节点独立是为了结果可信

管理节点的 CPU 压力一般低于施压节点，但它承担任务调度、WebSocket 实时进度、Redis Pub/Sub、PostgreSQL、报告读取、JTL 对比、PDF/HTML 报告等工作。压测结束后做报告分析时，管理节点会出现 CPU、内存和磁盘 I/O 峰值。

将管理节点和施压节点拆开，有三个价值：

1. 压测发流量不受报表查询、JTL 解析、Web 页面访问影响。
2. 施压机故障或重启不影响平台任务记录和历史报告。
3. 后续扩容时只增加施压节点，不需要频繁改管理节点。

## 4. 1-3 台部署拓扑

### 4.1 一台机器：开发/演示拓扑

```text
Linux-01
  Manager + Web UI
  Redis
  PostgreSQL
  Agent
  JMeter
  scripts / csv / reports / logs
```

适用场景：

- 平台演示。
- 脚本调试。
- 低并发冒烟。
- 非正式压测。

风险：

- JMeter 发压会和数据库、Redis、报告生成争抢资源。
- 同机资源瓶颈会影响压测结果可信度。
- 磁盘写入、CPU 打满时 Web 页面也会卡。

### 4.2 两台机器：生产起步拓扑

```text
Linux-01 管理节点
  Manager + Web UI
  Redis
  PostgreSQL
  scripts / csv / reports / logs

Linux-02 施压节点
  Agent
  JMeter
  local jtl / local logs
```

适用场景：

- 团队日常压测。
- 并发多数 500-800。
- QPS 多数 1000-1600，偶发接近 2000。
- 预算有限，但希望管理和发压隔离。

### 4.3 三台机器：推荐正式拓扑

```text
Linux-01 管理节点
  Manager + Web UI
  Redis
  PostgreSQL
  scripts / csv / reports / logs

Linux-02 施压节点 A
  Agent
  JMeter

Linux-03 施压节点 B
  Agent
  JMeter
```

适用场景：

- 正式平台部署。
- 需要给多个业务线复用。
- 日常并发低于 1000，但希望测试结论稳定。
- QPS 峰值可能接近 2000。
- 需要保留历史报告、做趋势分析和任务对比。

## 5. 配置明细

### 5.1 管理节点配置

| 配置项 | 起步值 | 推荐值 | 原因 |
|---|---:|---:|---|
| CPU | 4C | 8C | FastAPI、报告生成、JTL 解析、PDF/HTML 报告、数据库查询 |
| 内存 | 8GB | 16GB | Manager、Redis、PostgreSQL、系统缓存、报告解析需要余量 |
| 磁盘 | 300GB SSD | 500GB SSD/NVMe | 存放脚本、CSV、JTL、HTML 报告、日志、数据库 |
| 网络 | 1Gbps | 1Gbps+ | 接收 Agent 状态、结果、报告访问，内网低延迟 |
| 系统 | Linux x86_64 | Linux x86_64 | 生产部署稳定、便于 systemd 和内核参数调优 |

管理节点内存分配建议：

| 组件 | 建议内存预算 |
|---|---:|
| Manager/FastAPI | 1-2GB |
| Redis | 1-2GB，设置 `maxmemory` |
| PostgreSQL | 2-4GB，按实际数据量调整 |
| OS 文件缓存 | 2-4GB |
| 报告/JTL 解析峰值 | 2-4GB 余量 |

如果 PostgreSQL 与 Redis 都在管理节点上，16GB 会明显比 8GB 舒适。PostgreSQL 官方文档对独立数据库服务器给出的 `shared_buffers` 起点建议是系统内存的 25%，但本平台是混部场景，不建议直接给到过大；8GB 管理节点可先给 1GB，16GB 管理节点可给 2-4GB。

### 5.2 施压节点配置

| 配置项 | 起步值 | 推荐值 | 原因 |
|---|---:|---:|---|
| CPU | 8C | 8C/16C | JMeter HTTP 请求、断言、采样结果处理、CSV 参数化都吃 CPU |
| 内存 | 16GB | 16GB/32GB | JMeter 堆内存、系统缓存、Agent、日志写入需要余量 |
| 磁盘 | 100GB SSD | 200GB SSD/NVMe | 本地 JTL、JMeter 日志、错误日志、临时文件 |
| 网络 | 1Gbps | 1Gbps，响应体大时 10Gbps | 主要压测流量从施压节点发出 |
| JVM 堆 | 4-6GB | 6-8GB | 根据单机承载并发和脚本复杂度调整 |

每台施压节点建议只承载当前活跃任务的一部分压力，不要把机器长期压到 90% 以上。建议运行中观察：

| 指标 | 建议阈值 |
|---|---:|
| CPU 使用率 | 长时间低于 70% |
| 内存使用率 | 长时间低于 75% |
| JVM GC | 不出现频繁 Full GC |
| 磁盘 I/O | await 不持续升高 |
| 网络带宽 | 长时间低于 60%-70% |
| JMeter 本地错误 | 不出现连接池/端口/文件句柄类错误 |

## 6. JMeter 执行建议

### 6.1 必须使用非 GUI 模式

正式压测必须使用 JMeter CLI/非 GUI 模式。GUI 只用于录制、调试和脚本开发。平台当前通过 Agent 调用 JMeter 命令行执行，方向是正确的。

### 6.2 减少资源浪费

建议压测脚本遵守以下规则：

- 禁用或删除 `View Results Tree`、`View Results in Table` 等调试监听器。
- 使用 CSV 格式 JTL，不使用 XML 格式 JTL。
- 只保存必要字段，不保存完整响应体。
- 只在必要接口加断言，避免大量复杂断言。
- CSV 参数化数据使用文件读取，不在脚本中生成大量随机数据。
- 错误日志按需开启，正式长稳压时避免保存过大的响应内容。

这些规则比单纯加机器更重要。脚本不精简时，500 并发也可能把施压机打满；脚本精简时，同样机器可以稳定承载更高 QPS。

### 6.3 分布式节点注意事项

如果使用 JMeter remote/distributed 模式，需要注意：

- 控制端和远端 JMeter 版本必须一致。
- Java 版本建议一致。
- CSV、依赖 jar、插件、证书等文件要在每个施压节点对应路径存在。
- JMeter RMI 默认涉及 1099 和动态端口，生产环境建议固定 `server.rmi.localport`、`client.rmi.localport`，方便防火墙放行。
- 不要把 JMeter Engine 部署在被测应用服务器上，否则会抢占应用资源并污染测试结果。

## 7. Linux 系统参数建议

正式压测前建议在管理节点和施压节点都设置文件句柄上限；施压节点还需要关注本地端口范围和 TCP 连接回收。

### 7.1 文件句柄

`/etc/security/limits.conf`：

```conf
* soft nofile 65535
* hard nofile 65535
* soft nproc 65535
* hard nproc 65535
```

systemd 服务也要配置：

```ini
LimitNOFILE=65535
LimitNPROC=65535
```

Redis 官方文档也说明，Redis 的最大客户端连接数会受操作系统文件描述符限制影响；如果系统 soft limit 不够，实际可用连接数会被下调。

### 7.2 网络参数

`/etc/sysctl.conf`：

```conf
fs.file-max = 2097152
net.ipv4.ip_local_port_range = 1024 65535
net.ipv4.tcp_fin_timeout = 15
net.core.somaxconn = 65535
net.core.netdev_max_backlog = 65535
```

是否启用 `tcp_tw_reuse` 要按公司内核版本和网络规范评审后决定，不建议在不了解影响的情况下盲目开启。

### 7.3 时间同步

所有节点启用 NTP/chrony。压测报告、请求日志、Agent 心跳、趋势对比都依赖时间一致性。

## 8. 端口和网络放行

| 方向 | 端口 | 说明 | 暴露范围 |
|---|---:|---|---|
| 用户 -> 管理节点 | 8000 或反向代理 80/443 | Web UI/API | 公司内网 |
| Agent -> Redis | 6379 | 任务命令、心跳、结果 Pub/Sub | 仅施压节点到管理节点 |
| Manager -> PostgreSQL | 5432 | 数据库 | 建议仅本机或管理节点内网 |
| JMeter Controller -> Slave | 1099/1100+ | RMI/Slave | 仅管理/施压内网 |
| Slave -> Controller | 固定 client.rmi.localport 范围 | RMI 反向结果连接 | 仅管理/施压内网 |
| SSH 运维 | 22 | 部署和排障 | 运维网段 |

Redis、PostgreSQL 不建议暴露到公网，也不建议对所有内网网段开放。

## 9. 数据保留和磁盘规划

建议默认策略：

| 数据类型 | 建议保留 |
|---|---:|
| HTML 报告 | 90 天 |
| JTL 原始文件 | 30-60 天 |
| 错误响应日志 | 7-30 天，按敏感等级控制 |
| Manager/Agent 日志 | 30 天 |
| CSV 数据文件 | 按项目归档，定期清理无用文件 |

如果每天都有多次 30-60 分钟压测，管理节点 300GB 会偏紧，500GB 更稳。后续可增加对象存储或 NAS 做报告归档，但不建议把 JMeter 实时写入目录直接放到慢速网络盘上。

## 10. 采购/汇报口径

可以对领导这样说明：

> 这个平台不是普通 Web 系统，核心价值是稳定地产生压力并保证测试结论可信。日常虽然并发很少超过 1000、QPS 很少超过 2000，但压测过程中施压机同时承担线程调度、HTTP 连接、响应处理、断言、结果落盘和报告生成。若管理服务和发压服务混在一台机器上，机器自身瓶颈会污染测试结果，导致我们无法判断到底是业务系统慢，还是发压机撑不住。  
>  
> 因此建议采用 1 台管理节点 + 2 台施压节点。管理节点负责平台、数据库、Redis、报告和历史数据；施压节点专门发流量。这样可以把 1000 并发/2000 QPS 分摊到两台施压机，每台机器只承担约一半压力，保留资源余量，测试结果更稳定。后续如果业务增长，不需要重构平台，只要继续增加施压节点即可。

## 11. 最终建议

如果只选一个方案，建议采购/申请：

| 机器 | 配置 |
|---|---|
| 管理节点 1 台 | 8C/16GB/500GB SSD/NVMe/1Gbps |
| 施压节点 2 台 | 每台 8C/16GB/200GB SSD/NVMe/1Gbps |

如果预算有限，最低生产可先申请：

| 机器 | 配置 |
|---|---|
| 管理节点 1 台 | 4C/8GB/300GB SSD |
| 施压节点 1 台 | 8C/16GB/200GB SSD |

扩容优先级：

1. 压测时施压机 CPU、网络、JVM GC 成为瓶颈：优先增加施压节点。
2. 报告生成慢、JTL 对比慢、页面查询慢：升级管理节点到 8C/16GB 或增加磁盘 I/O。
3. 历史报告增长快：增加管理节点磁盘或接入归档存储。
4. 响应体很大或跨机房压测：优先优化网络位置，再考虑 10Gbps。

- Apache JMeter Best Practices：官方建议用 CLI 模式、减少 Listener、不要在负载测试中使用 View Results Tree/Table、使用 CSV 输出、只保存必要数据。  
  https://jmeter.apache.org/usermanual/best-practices.html
- Apache JMeter Remote Distributed Testing：官方说明远程分布式测试需要多台 JMeter Engine、版本一致、数据文件在各节点可用，并注意 RMI 端口和防火墙。  
  https://jmeter.apache.org/usermanual/remote-test.html
- PostgreSQL Resource Consumption：官方建议独立数据库服务器 `shared_buffers` 可从系统内存 25% 起步，但不宜过度占用 OS cache。  
  https://www.postgresql.org/docs/current/runtime-config-resource.html
- Redis Memory Optimization / Client Handling：官方说明 `maxmemory`、峰值内存、`maxclients` 和 OS 文件描述符限制之间的关系。  
  https://redis.io/docs/latest/operate/oss_and_stack/management/optimization/memory-optimization/  
  https://redis.io/docs/latest/develop/reference/clients/
